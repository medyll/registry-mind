## Story Details

### S5-01 ŌĆö Fix double-capture bug
**Problem:** `CaptureService.startCapture()` calls `captureOnly()` ŌåÆ shows VetoBar ŌåÆ on commit calls `captureAndProcess()` which internally calls `captureOnly()` AGAIN. Second screenshot may differ from the one the user approved.

**Fix:**
- Add `suspend fun processPacket(packet: RegistryPacket)` to `RegistryIngestor` (extracts the logic from `processAndStore` ŌĆö or makes it internal/public)
- `CaptureService.startCapture()` on commit: call `ingestor.processPacket(packet)` with the original packet

**AC:**
- [ ] `RegistryIngestor` has public `suspend fun processPacket(packet: RegistryPacket)`
- [ ] `CaptureService` on commit calls `processPacket(packet)` not `captureAndProcess()`
- [ ] `captureAndProcess()` still works for non-veto flows (e.g. direct trigger)

---

### S5-02 ŌĆö LocalLlm init in CaptureService
**Problem:** `CaptureService.onCreate()` creates `RegistryIngestor` but never sets `ingestor.localLlm`. All captures fall back to `rawText.take(200)` summary.

**Fix:**
- On `CaptureService.onCreate()`, after RegistryIngestor creation: check `ModelDownloadManager.isDownloaded`
- If yes: launch coroutine on Dispatchers.Default ŌåÆ `LocalLlm(context).init(modelFile)` ŌåÆ assign to `ingestor.localLlm`
- If no: leave null (fallback active), schedule model download notification

**AC:**
- [ ] `CaptureService` creates `LocalLlm` and calls `init()` when model file exists
- [ ] Assign to `ingestor.localLlm` after successful init
- [ ] Close `LocalLlm` in `onDestroy()`
- [ ] No crash if model file missing (null path)

---

### S5-03 ŌĆö First-launch model download
**Problem:** `ModelDownloadManager` exists but nothing calls `download()`. Gemma model never gets to the device.

**Fix in MainActivity:**
- On `onCreate`: check `ModelDownloadManager.isDownloaded`
- If not: show a full-screen `ModelDownloadDialog` (composable) with: "AI model required (~1.3 GB)", Progress bar, Cancel button
- Use `MODEL_URL` constant (configurable via SettingsManager or hardcoded for now)
- On complete: dismiss dialog, show "Model ready" snackbar

**AC:**
- [ ] `MainActivity` checks model on start
- [ ] Dialog shown when model missing, dismissed when complete or cancelled
- [ ] `LinearProgressIndicator` updates during download via `onProgress` callback
- [ ] Download runs on `Dispatchers.IO` (no ANR)
- [ ] No duplicate download if already in progress

---

### S5-04 ŌĆö EntriesScreen
**Problem:** Users can't see their captured entries anywhere. The main screen only shows service status.

**Fix:**
- Add a "Captures" tab or button in `MainActivity` leading to `EntriesScreen`
- `EntriesScreen`: `LazyColumn` of `EnrichedEntryCard` composables
- Each card: summary (2 lines), tag chip (Urgent/Personal/Work), sourceApp (small), timestamp (relative)
- Data: `collectAsStateWithLifecycle` on `CacheDatabase.enrichedEntryDao().observeAll()`
- Empty state: "No captures yet ŌĆö press Snap Key to start"

**AC:**
- [ ] `EntriesScreen` composable in `MainActivity.kt` (or separate file)
- [ ] `EnrichedEntryCard` shows summary, tag, sourceApp, timestamp
- [ ] Empty state rendered when list is empty
- [ ] Real-time updates via Flow (new captures appear without restart)
